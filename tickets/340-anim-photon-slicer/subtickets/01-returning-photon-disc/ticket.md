# Re-theme Photon Slicer into a returning photon-disc boomerang synced to its server effect

Photon Slicer's server effect is `returning_projectile` (a thrown disc that flies out to `attackRange` and boomerangs home — icon `⟲`, evolves into "Infinite Disk"), but the client currently renders it with `renderWeaponSwing` as a **stationary melee cone slash** (`coneAngle: Math.PI`). Replace that with a dedicated single spinning cyan photon-disc throw-and-return renderer (the single-disc sibling of the existing `renderTripleReturning` used by `infinite_disk`), so the animation reads unmistakably as "Photon Slicer" and its outbound+return beats line up with the server's synchronous `returning_projectile` resolution.

## Acceptance Criteria
- `photon_slicer` is registered in `CARD_RENDERERS` to a dedicated returning-disc renderer (a new function, e.g. `renderReturningDisc`), NOT `renderWeaponSwing` and NOT `renderConeSwings`.
- The new renderer spawns the outbound disc visual via `ctx.spawnAttackEffect(origin, direction, …)` tinted with Photon Slicer's cyan accent (`getAccentHex('photon_slicer')` → `0x22d3ee`, falling back to a cyan default such as `0x22d3ee`/`0xa5f3fc` if the accent is absent).
- The disc travels the weapon's reach taken from the payload: it reads `data.attackRange` (Photon Slicer emits `attackRange: 8`) and falls back to a finite default when absent, instead of a hardcoded cone range. A forward trail (`ctx.spawnProjectileTrail`) and a spark burst (`ctx.spawnParticleBurst`) are emitted along/at the far point of that outbound path.
- The renderer plays at least one boomerang **return beat**: it schedules a return-pass via `ctx.scheduleAfter` whose trail/burst travels from the far point back toward the origin. The number of return beats follows the payload's `returnPasses` but **defaults to 1** when `returnPasses` is omitted (Photon Slicer's payload omits `returnPasses`, yet the server runs exactly one return pass) — it must NOT default to 0 (the `?? 0` behaviour used by `renderTripleReturning` would leave the base card with no visible return).
- The return-beat cadence is derived from `ATTACK_EFFECT_DURATION` (e.g. a fraction of it, as `INFINITE_DISK_RETURN_BEAT_MS` does), so the full throw+return flourish resolves within the attack-effect window and stays in sync with the server's same-tick hit resolution — no fixed multi-second delay.
- The renderer degrades gracefully: calling it when `spawnProjectileTrail`, `spawnParticleBurst`, and `scheduleAfter` are all absent does not throw (every optional primitive is guarded).
- The now-unused `photon_slicer` entry in `WEAPON_SLASH_STYLES` is removed (it is dead config once `photon_slicer` no longer routes through `renderWeaponSwing`); entries for other weapons are left untouched.
- Client tests are updated and pass: the existing `'Photon Slicer cuts a wide cyan spin slice with a light trail'` test (which asserts the melee `coneAngle: Math.PI` swing) is replaced by a test asserting the disc throw (cyan `spawnAttackEffect` along the firing direction at `attackRange`, a `spawnProjectileTrail`, a far-point `spawnParticleBurst`) plus a scheduled return beat (a `scheduleAfter` entry that, when run, emits a return trail/burst). The "five energy blades use mutually distinct accent colors", "degrade gracefully", and "card-specific renderers … (not the cone default)" tests still pass with `photon_slicer` included.
- `pnpm test` (server + client vitest) passes with no perf regression; no files outside the Technical Specs list are changed.

## Technical Specs

Compute return-pass count as `Math.max(1, data.returnPasses ?? 1)` so the base card always shows one return.
