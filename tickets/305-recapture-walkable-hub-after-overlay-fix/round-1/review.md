## Harness blockers

The review capture could not produce a clean run because the harness chose its
generic fallback plan, and that plan contains a stale, self-contradictory
assertion. Neither is in this ticket's scope and neither is game code.

1. **Wrong plan selected.** `metrics.json` shows
   `"capturePlanSource": "fallback"` and `"scenarios": ["telepipe-ready"]`, with
   `capturePlanSummary` = *"Deterministic solo Telepipe suspend/resume
   capture …"*. The review machinery ran a solo telepipe suspend/resume capture,
   **not** the walkable-hub capture this ticket is about — the captured frames
   are `01-in-dungeon / 02-suspended-lobby / 03-resumed-dungeon`.

2. **The fallback assertion contradicts current game design.**
   `assertRunPreserved` (`harness/screenshot.mjs:1116-1199`) expects telepipe
   suspend→resume to *restore a checkpoint* (same enemy IDs, same objective).
   The current post-287 game intentionally does a **fresh redeploy**: the
   resumed run has a new `runId`, a regenerated dungeon (same seed 352369970,
   profile `crowded`, 10 rooms) and new enemy instance IDs, preserving only
   vitals. The sibling `assertVitalsPreserved` (`harness/screenshot.mjs:1066`)
   even *requires* `freshRunId` (runId must differ). So `assertRunPreserved` can
   never pass against the current game and fails on every capture regardless of
   this ticket. The evidence confirms it: `preservation.preservedIds = 0`, all 5
   pre-suspend IDs in `missingIds`, 5 fresh IDs in `addedEnemies`, and
   `objective: null` (the suspended-state probe reports `objective: null`).

Because the game ran cleanly (no page errors, servers up, dungeon playable) and
the failing code is the harness's own review-capture machinery — not game code
and not this ticket's diff — further edits to `game/` or to the hub validation
driver will NOT change this outcome.
