# Render the equipped key item as a body prop on the avatar

Map each player's equipped key item (the snapshot's `equippedKeyItemId`) to a small, distinct prop/badge mesh anchored to the avatar body, mirroring the existing hat feature (`buildHatMesh` / `attachGltfHat`). The prop seats on a glTF body bone (with a procedural-fallback prop and a no-bone fallback), updates when the equip changes, renders for both local and remote players, and renders nothing for key items without a defined prop.

## Acceptance Criteria
- A new builder `buildKeyItemProp(keyItemId)` returns a distinct `THREE.Object3D` for key items that have a defined prop, and returns `null` for `'none'`, `null`/undefined, and any unknown/unmapped id (graceful "no prop").
- At least four catalog key items get visibly distinct props (e.g. `dodge_roll`, `guard_block`, `loot_magnet`, `field_medic_kit`), each with its own geometry and color; ids without a mapping fall through to `null` (no prop) rather than erroring.
- On the loaded glTF avatar, the prop is parented to a body bone — preferring `spine_03`, then `spine_02`, then `spine_01` — with world-scale and orientation compensation so it reads as worn on the torso and rotates with the avatar (analogous to `attachGltfHat`). When no spine bone exists it falls back to a host-local chest anchor so the prop still renders.
- The procedural primitive-fallback avatar also gets the prop seated on its torso (so the prop exists even before/without the glTF), and that procedural prop is hidden alongside the procedural body when the glTF model resolves (same handling as the hat).
- The rendered prop updates when a player's `equippedKeyItemId` changes between snapshots WITHOUT a page reload: the old prop is removed from its parent and disposed, and the new prop is built and re-attached. The currently-shown id is tracked on `host.userData.keyItemId`.
- Props render for BOTH the local player and remote players (the equip-change/attach logic runs in both the remote-player branch and the local-player branch of the render-loop player update).
- Unknown / asset-less / `'none'` key items add no prop and never throw; the whole prop-attach routine is best-effort and caught (no crash, no removed avatar).

## Technical Specs
- `game/client/renderer.js` (no server changes — the snapshot already exposes `equippedKeyItemId` per player at `game/server/progression.js:3171`):
  - Add a key-item prop catalog and `buildKeyItemProp(keyItemId)` next to `buildHatMesh` (~line 1365). Mirror `buildHatMesh`'s structure: a `switch` over supported ids returning a `THREE.Group`/`THREE.Mesh` with distinct geometry + `MeshStandardMaterial` color per id; `case 'none'`/`default` returns `null`. Key item ids come from the server `KEY_ITEM_DEFS` (`game/server/progression.js:565`).
  - Add `attachGltfKeyItemProp(host, model)` next to `attachGltfHat` (~line 471). Resolve the body bone via `model.getObjectByName('spine_03') || model.getObjectByName('spine_02') || model.getObjectByName('spine_01')`; parent the prop to it with the same world-scale/quaternion compensation pattern `attachGltfHat` uses, plus a small chest offset. If no bone is found, `host.add(prop)` at a fixed local chest position. Store the result on `host.userData.keyItemPropMesh`. Call it in the `key === 'player'` branch of `attachRegistryModel` (~line 387) AFTER `attachGltfHat(host, model)` so it is built after the procedural-mesh visibility snapshot and is not hidden.
  - In `createPlayerAvatar` (~line 1481): build the procedural prop for the resolved equipped id, seat it on the body torso, add it to the group, and record `group.userData.keyItemId` (the id used to build it) and `group.userData.keyItemPropMesh`. The procedural prop is hidden with the body when the glTF resolves (it is part of the procedural snapshot in `attachRegistryModel`, same as the body — confirm it is captured by that snapshot loop, not excluded like the hat).
  - In the render-loop player update, add prop equip-change handling for both branches: the remote-player loop (~lines 3470-3508) and the local-player block (~lines 3511+). When `pData.equippedKeyItemId !== playersMeshes[id].userData.keyItemId`, remove the old `userData.keyItemPropMesh` from its parent, dispose it, build a new prop via `buildKeyItemProp`, attach it (to the glTF body bone when `userData.modelOverride` is present, else to the procedural group/torso), and update `userData.keyItemId` and `userData.keyItemPropMesh`.
  - Reuse the existing constants/patterns from the hat code (world-scale factor, offset, fallback Y) as the model for the prop's analogous constants.
- Add or extend a client test under `game/client/test/` (e.g. `playerModel.test.js`) asserting `buildKeyItemProp` returns a non-null object for a mapped id (e.g. `dodge_roll`) and `null` for `'none'` and an unknown id.
